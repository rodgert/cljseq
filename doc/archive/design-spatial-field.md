# Design: Spatial Field

*Generalizing Bitwig's Ricochet Note FX — from "notes as balls in a room" to a physics-simulation parameter manifold that drives generation, modulation, and spatial routing.*

---

## The Generalization

Ricochet's model: notes are balls launched into a bounded room. Collisions with walls or other balls fire new note events. Ball position drives pan (X) and timbre (Y).

The generalization: **a Spatial Field is a 2D bounded manifold where particles travel under configurable physics, and their positions continuously drive any combination of musical output parameters**. Pan is just one possible mapping — the axes of the space can map to anything.

```
Space = {
  bounds,        ; room geometry
  particles[],   ; active entities with position, velocity, mass
  physics,       ; forces: damping, gravity, attractors, repellers
  mappings[],    ; (axis → musical parameter) × N
  behaviors[],   ; what happens on collision/boundary events
}
```

---

## The Room Geometry Insight

Bitwig exposes Room Sides as a continuous parameter from 3.0 to 8.0, including decimals.

This is not cosmetic. The number of sides determines the reflection angles, which determine the inter-collision timing intervals, which produce the rhythm:

| Room Sides | Reflection geometry | Rhythmic character |
|---|---|---|
| 3.0 | Equilateral triangle | Triplet-like |
| 4.0 | Square | Regular pulse (equal intervals) |
| 5.0 | Pentagon | Quintuplet / pentatonic-adjacent |
| 6.0 | Hexagon | Whole-tone subdivisions |
| 7.0 | Heptagon | 7-against-4 feel |
| 8.0 | Octagon | Dense / subdivision-rich |
| 3.5 | Asymmetric tri/square hybrid | Irregular / polyrhythmic |
| 4.7 | Near-square asymmetric | Complex irregular pulse |

**Room geometry encodes rhythmic structure through reflection geometry.** A Spatial Field is not just a physics toy — it is a rhythmic structure generator where the shape of the space is the compositional parameter.

An asymmetric room (non-integer sides) produces incommensurate reflection intervals — the same mathematical territory as `phase-pair` in the Berlin School design, where LCM analysis of incommensurate loop lengths creates slowly evolving polyrhythmic texture. Here the "loops" are particle paths through an asymmetric space.

## Three Operating Modes

### Mode 1: Generation (Ricochet analog)

Particles are launched when note events arrive. Collisions between particles, or between particles and walls, **fire new note events**.

The launched particle's initial velocity is scaled from the triggering note's velocity. Ball Damping reduces speed on each collision — fast balls generate more collisions (denser rhythm); slow balls generate fewer (sparser). When a ball stops moving (velocity below threshold), it despawns or enters a hold state.

**Room Spatialization in Generation mode**: Each new note's pan position is derived from the particle's position at the moment of collision. Notes generated by wall collisions on the left have left pan; right wall → right pan; top/bottom wall → timbre dimension. The room becomes a spatial palette for generated note placement.

```clojure
(defspatial-field rhythm-room
  :mode      :generation
  :sides     4.7               ; near-square asymmetric
  :ball-speed    1.0           ; velocity-relative launch speed
  :ball-damping  0.85          ; 85% speed retention per collision
  :ball-radius   0.1
  :min-velocity  0.05          ; despawn below this speed
  :mappings  {:x :pan          ; left-right position → stereo pan
              :y :timbre})     ; up-down position → MPE timbre
```

### Mode 2: Modulation

Particles travel independently of incoming note events — they move continuously through the space under physics, and their positions drive output parameters as **continuously updated control values** (equivalent to LFOs, but with physics-driven trajectories).

Unlike an LFO (periodic, mathematically predictable) or a Trajectory curve (deterministic), a Spatial Field in Modulation mode produces trajectories that are:
- Bounded (particles stay in the room)
- Sensitive to initial conditions (chaotic in asymmetric rooms)
- Self-interacting (multiple particles affect each other)
- Ergodic over long time scales (will eventually visit all areas, but not periodically)

This fills a gap: something between the predictability of phasors/trajectories and pure randomness. Useful for ambient textures where slow parameter drift feels organic without being mechanically periodic.

```clojure
(defspatial-field slow-pan-drift
  :mode      :modulation
  :sides     5.3               ; irregular pentagon
  :particles 2                 ; two slow-moving particles
  :ball-speed    0.1           ; slow drift
  :ball-damping  1.0           ; no slowdown (elastic bounce)
  :mappings  {:x :pan          ; continuous stereo position
              :y :reverb-send  ; up-down = wet/dry
              :r :velocity})   ; distance from center = velocity
```

**Mapping :r (radial distance from center)**: A particle near the center is "in the room"; near the walls is "at the edge." Distance from center maps naturally to reverb send (close/dry ↔ far/wet), velocity, or filter cutoff.

### Mode 3: Routing

Particle position determines which output destination an event is sent to. The space is divided into zones (Voronoi regions around output targets), and events generated or passing through the Spatial Field are routed to the zone they currently occupy.

```
Zone map example (4 zones for 4 instruments):
  Top-left     → kalimba
  Top-right    → strings
  Bottom-left  → bass
  Bottom-right → percussion
```

As a particle moves through the space, the notes it generates or carries shift between instruments — creating a spatial composition where the arrangement of instruments is encoded in the geometry of the space.

```clojure
(defspatial-field instrument-space
  :mode      :routing
  :sides     4.0               ; square — four clean quadrants
  :zones     {:top-left    :kalimba
              :top-right   :strings
              :bottom-left :bass
              :bottom-right :percussion}
  :blend     0.2)              ; crossfade within 20% of zone boundary
```

The `:blend` parameter creates smooth handoffs at zone boundaries — notes near a zone edge may be sent to both adjacent instruments at blended velocities, rather than hard-switching.

## Physics Behaviors

### Forces

```clojure
:forces {
  :gravity    {:direction :down :strength 0.3}  ; constant pull
  :attractor  {:position [0.5 0.5] :strength 0.5 :falloff :inverse-square}
  :repeller   {:position [0.0 0.0] :strength 0.8 :falloff :linear}
  :wind       {:direction :right :strength 0.1}  ; constant drift
}
```

**Attractor**: Particles are pulled toward a point. A central attractor with moderate strength + damping creates spiral-in trajectories. Multiple attractors create complex multi-lobe paths.

**Gravity well at center + elastic walls**: Particles orbit with perturbation from wall reflections — produces quasi-orbital paths, never quite repeating (like a perturbed planetary orbit).

**Repeller at center**: Particles avoid the center, clustering near walls — useful in Routing mode to keep events near the instrument zones rather than crossing through a "dead" center.

### Boundary Behaviors

| Boundary type | On collision |
|---|---|
| `:reflect` | Elastic reflection (angle in = angle out) |
| `:absorb` | Particle despawns; note-off if in Generation mode |
| `:portal` | Particle re-enters from opposite wall (wrapping geometry) |
| `:generate` | Fires a new note event (Ricochet default) |
| `:attract` | Wall pulls particle back (curved reflection) |

Mixed boundary behaviors per wall enable asymmetric rooms: left wall reflects, right wall generates, top absorbs, bottom portals. Each combination produces a different musical character.

### Particle Properties

```clojure
{:position [x y]          ; current position in [0.0, 1.0]²
 :velocity [vx vy]        ; current velocity
 :mass     1.0            ; affects force response (lighter = more reactive)
 :charge   1.0            ; +/- for attractor/repeller interaction
 :age      0              ; ticks since launch
 :note     event}         ; the triggering event (in Generation mode)
```

**Charge**: Positive particles are attracted to attractors and repelled by repellers. Negative charge inverts both. Mixing positive and negative particles creates grouping behavior — positive particles cluster near attractors while negative ones cluster at walls.

## Connection to Studio Spatial Topology

The `project_device_group_spatial.md` memory note identifies the spatial layer (pan/level/routing) as a trajectory target — "stereo pair generalizes with programmable mixer; spatial layer becomes trajectory target."

The Spatial Field in Modulation mode is the physical-simulation implementation of spatial trajectories:
- **Trajectory** (design-temporal-buffer.md): deterministic curve through parameter space
- **Phasor**: periodic sweep
- **Spatial Field**: physics-driven, bounded, self-interacting, ergodic

In the studio topology context, a Spatial Field controlling `:pan` and `:reverb-send` for a device group produces the kind of slowly evolving spatial placement that is currently only achievable by hand-automating a mixer. With a Spatial Field, the space itself is the automation.

**Physical room ↔ virtual room correspondence**: For a studio with physical speaker placement (stereo, quad, or Atmos), the Spatial Field's X axis could map to real speaker position. A particle that bounces to the left wall sends audio to the left speaker; a particle crossing to the right sends to the right. The room simulation produces sound spatialization that follows the same physics as the note generation — the musical geometry and the acoustic geometry are unified.

## Connection to Other cljseq Designs

**Threshold Extractor** (design-threshold-extractor.md): The Extractor fires events when a continuous value crosses a threshold. A Spatial Field's X or Y position is a continuous value — an Extractor watching a particle's X position fires an event each time it crosses the center line. The wall collision in Generation mode is a special case of threshold crossing at the boundary.

**Temporal Buffer** (design-temporal-buffer.md): A Spatial Field in Generation mode can feed a Temporal Buffer — generated collision-notes are recorded, then replayed, re-analyzed by the Spectral Resynthesizer. Each layer adds harmonic richness derived from the physics.

**Spectral Resynthesizer** (design-spectral-resynthesizer.md): Spatial Field modulation → drives the Spectral Resynthesizer's Slide parameter (scanning through the Harmonic Library). The particle position selects which stored harmonic snapshot is active — physics drives harmonic morphing.

**Ensemble Improvisation**: In a multi-loop ensemble, each loop could have its own particle in a shared Spatial Field. Their positions relative to each other drive cross-loop dynamics (a particle moving toward another → call and response; two particles near the same wall → harmonic convergence).

## The Echo/Dribble Unification

Bitwig's Echo (time + pitch decay) and Dribble (gravity-physics timing) are both special cases of a 1D Spatial Field:

**Dribble as 1D Spatial Field**:
- Y axis only; X is fixed
- Gravity force: `:gravity {:direction :down :strength g}`
- Wall behavior: `:reflect` at Y=0 (floor), `:absorb` at Y=1 (ceiling)
- Damping: velocity loss per bounce = `damping`
- Collision → generates note event
- Shortest bounce = minimum inter-event time threshold

**Echo as 1D Spatial Field with pitch drift**:
- Y axis = pitch space; X = time
- Particle moves at constant velocity in X (time advances)
- Walls reflect in Y (pitch bounces between bounds)
- Velocity decay in Y → diminuendo
- Each wall collision fires a note at current Y position (current pitch)
- Result: a note echo that drifts in pitch as if bouncing between pitch walls

This unification suggests that Dribble, Echo, and Ricochet should all be *named configurations* of the Spatial Field — just as Flux, Memory Field, and Note Mimeophon are named presets of the Temporal Buffer.

## The Third Dimension: Z as Acoustic Depth

The 2D field maps X (pan) and Y (any musical dimension). Adding Z unlocks a third axis — and the most physically grounded meaning for Z is **acoustic depth**: distance from the listener.

This is not an arbitrary choice. In real acoustic space, distance bundles a set of parameters that naturally co-vary:

| Distance increases | Effect |
|---|---|
| → quieter | Inverse-square attenuation |
| → more reverb | Room effect dominates over direct signal |
| → longer pre-delay | Sound takes time to travel |
| → darker (HF rolloff) | Air absorbs high frequencies over distance |
| → temporal smear | Diffuse reflections blur the transient |

These five effects are currently separate parameters in cljseq. Z collapses them into a single physical metaphor — one knob, coherent physical behavior.

```clojure
;; Z=0 → dry, loud, immediate, bright, precise
;; Z=1 → wet, quiet, delayed, dark, smeared
(map-axis :z [:velocity      {:range [1.0 0.3]}   ; attenuation
              :reverb-send   {:range [0.0 1.0]}   ; room effect
              :delay-pre     {:range [0   80]}    ; ms pre-delay
              :filter-cutoff {:range [1.0 0.4]}   ; HF rolloff
              :halo          {:range [0.0 0.8]}]) ; temporal smear
```

### Z as Temporal Depth — The Space-Time Unification

There is a deeper interpretation: **Z = position in the Temporal Buffer**.

- Z=0 → now (most recent events)
- Z=1 → deepest past (oldest buffer zone)

A particle at Z=0 plays current events; at Z=1 it reads from the deepest Temporal Buffer zone. **Moving along Z is moving through time.** The Temporal Buffer's 8 discrete zones become a continuous Z axis — the Spatial Field in 3D contains the Temporal Buffer as one of its dimensions.

This is the acoustic metaphor made precise: sound "in the distance" (high Z) is also sound "from the past" — it took longer to reach you. The physical and temporal metaphors are the same metaphor.

In Generation mode with Z=temporal depth: a particle launched into the room and moving away (increasing Z) is generating notes drawn from progressively older buffer zones. As it comes back (decreasing Z) it plays progressively more recent events. The room is a time machine.

In Modulation mode: Z controls the Temporal Buffer's read head position — the particle's path through Z sweeps the read head across the buffer, creating a physically-motivated Slide (comparable to the Spectral Resynthesizer's Slide parameter scanning the Harmonic Library).

### 3D Room Geometry

In 2D, room shape is an N-gon (3–8 sides, including decimals). In 3D, room shape is a **polyhedron**:

| Shape | Faces | Character |
|---|---|---|
| Tetrahedron | 4 | Very sparse; large intervals between collisions |
| Cube | 6 | Structured; 3 orthogonal reflection planes; regular |
| Octahedron | 8 | Moderate; 4 symmetry axes |
| Dodecahedron | 12 | Rich; quasi-irregular from most launch angles |
| Icosahedron | 20 | Complex; near-ergodic |
| Sphere | ∞ | Perfectly uniform; maximum ergodicity; all regions visited equally |
| Irregular convex hull | N | Arbitrary; encode specific polyrhythmic structures by face count + angles |

**Corner reflections** are qualitatively new in 3D. A ball hitting an edge gets two simultaneous reflections (one per adjacent face). A corner gets three. Corners produce **triply-reinforced arrival times** — concentrated local bursts of collisions. In Generation mode, corners are rhythmic hotspots: they fire dense clusters of notes at predictable geometric positions. A cube's 8 corners become compositional landmarks.

**Sphere room** is the limiting case: all reflection angles are equal, all positions are visited with equal probability over time. The sphere is the most homogeneous modulation source — no rhythmic structure, pure ergodic coverage of the parameter space.

**Irregular convex hulls** (3D equivalent of decimal room sides): by specifying face normals rather than a named polyhedron, you can encode arbitrary reflection geometry. This is the highest-level compositional tool — the hull shape *is* the rhythmic and spatial structure of the piece.

### Z as Harmonic Register

Vertical position (Z) maps naturally to pitch register — how we perceive sound spatially:

- High Z → treble, upper harmonics, "above" (bright, aerial)
- Mid Z → mid register (presence, vocal range)
- Low Z → bass, sub-octave, "below" (dark, foundational)

Combined with the Spectral Resynthesizer's odd/even harmonic routing:
- High Z → odd partials (hollow, bright, clarinet-like)
- Low Z → even partials (octave reinforcement, full, warm)

A particle drifting vertically in a 3D room produces natural register drift — the melody rises and falls not because a sequence says so, but because the physics of a gravitating particle pulls it downward, and rebounds send it upward.

### Z for Physical Spatial Audio Routing

For studios with actual height speakers — Atmos 7.1.4, quad + overhead, or even a simple stereo pair + ceiling bouncer — Z becomes a real physical routing dimension:

```
X = left/right  →  FL/FR/SL/SR channel selection + pan
Y = front/back  →  front/surround pair selection
Z = height      →  floor/overhead channel selection
```

The Spatial Field becomes a physical spatial audio router: particle position in the virtual room maps to actual speaker positions in the studio. The room simulation produces sound spatialization that follows the same physics as the note generation — the musical geometry and the acoustic geometry are unified.

With the studio topology (RME Digiface as hub, ExpertSleepers CV bridge for Eurorack integration), Z-routing could drive physical CV outputs that control a hardware mixer's spatial sends — the Spatial Field driving analog routing through the Eurorack.

## Quadraphonic + Binaural Output Architecture

Physical multichannel output is a near-term realizable goal that does not require Atmos infrastructure. A quad speaker array arranged around the modular rig — four speakers at the corners — is sufficient for immersive spatial diffusion, and it is a well-established format in the electronic/experimental music tradition (Stockhausen's diffusion systems, acousmatic concerts, modular synthesis performances).

### Quad Speaker Array

```
        FL ●───────────● FR
           │           │
           │  modular  │
           │   rig     │
        RL ●───────────● RR
```

Four speakers: FL (front-left), FR (front-right), RL (rear-left), RR (rear-right). The performer/listener is at the center of the array.

Speaker positions in normalized space:
```
FL = (-1, +1)    FR = (+1, +1)
RL = (-1, -1)    RR = (+1, -1)
```

### VBAP Amplitude Panning

For a source at position (x, y) in [-1, 1]², compute per-channel gains using vector-based amplitude panning (VBAP). cljseq computes these gains directly — no external panning software required.

Constant-power decomposition:

```clojure
(defn quad-gains
  "Compute [FL FR RL RR] amplitude gains for position [x y] ∈ [-1,1]².
   Returns gains summing to constant power (sum-of-squares = 1.0)."
  [[x y]]
  (let [;; front-back split: y=+1 → all front, y=-1 → all rear
        front (Math/sqrt (* 0.5 (+ 1.0 y)))
        rear  (Math/sqrt (* 0.5 (- 1.0 y)))
        ;; left-right split within each pair
        left  (Math/sqrt (* 0.5 (- 1.0 x)))
        right (Math/sqrt (* 0.5 (+ 1.0 x)))]
    [(* front left)   ; FL
     (* front right)  ; FR
     (* rear  left)   ; RL
     (* rear  right)])) ; RR
```

For 3D positions (x, y, z), elevation can additionally route to a center overhead channel or apply a height coefficient:

```clojure
;; Z ∈ [0,1]: 0 = floor-level (full quad), 1 = ceiling (blend toward mono overhead)
(defn quad-gains-3d [[x y z]]
  (let [base (quad-gains [x y])
        overhead-blend z]
    (mapv #(* % (- 1.0 overhead-blend)) base)))
```

### OSC Mixer Control Path

```
cljseq Spatial Field
   → per-tick particle position (x, y)
   → quad-gains [FL FR RL RR]
   → OSC /mixer/channel/1/gain  FL-value
      OSC /mixer/channel/2/gain  FR-value
      OSC /mixer/channel/3/gain  RL-value
      OSC /mixer/channel/4/gain  RR-value
   → OSC-controlled mixer
   → 4 amplifier channels → 4 speakers
```

The OSC message schema is mixer-agnostic. Concrete targets:
- **RME TotalMix FX** — limited OSC, but MIDI CC remote is well-supported; cljseq can send CC instead of OSC for TotalMix gain control
- **Bitwig** (Ubuntu DAW) — OSC remote control; quad channel faders on a track group
- **SuperCollider** — `Synth(\vbap4, [\fl, fl-gain, \fr, fr-gain, ...])` over OSC; SC has native quad panning (PanB2, Rotate2, etc.)
- **Custom clj bridge** — a lightweight process on the Mac Mini that receives OSC from cljseq and writes to the audio driver's channel gains directly

SuperCollider is the most natural target because its audio bus routing, multi-channel panning, and OSC interface are designed for exactly this. The `cljseq.spatial → OSC → SC` path means SC handles the audio path (quad VCA matrix) while cljseq owns the spatial model and physics.

### ExpertSleepers CV Feedback Path

The ExpertSleepers ES-3/ES-5/ES-6 trio bridges between the ADAT lightpipe network and Eurorack CV. The ES-3 converts ADAT channels to CV outputs; the ES-5 piggybacks onto the ES-3 and derives 8 gate outputs from two of the ES-3's ADAT channels (leaving the remaining 6 for CV); the ES-6 converts CV/gate inputs back to ADAT channels. CV and gate signals from the modular appear as audio channels on the computer via the ADAT chain.

**Inbound CV → cljseq:**

```
Modular CV source (joystick, LFO, envelope follower, gate)
   → ES-6 CV input (DC-coupled, typically ±5V normalized) → ADAT → Digiface
   → macOS Core Audio stream (audio channel with DC content)
   → cljseq sidecar or bridge process
      reads DC level of "audio" channel
      maps ±5V → [0.0, 1.0]
   → OSC /spatial/x  value
      OSC /spatial/y  value
      OSC /spatial/physics/speed  value
   → cljseq.ctrl receives → updates Spatial Field live
```

This gives the modular direct hands-on control over the spatial model:
- Joystick X/Y → particle position or launch angle
- LFO → ball speed or room damping (slow LFO = evolving room size)
- Envelope follower from a drum machine → particle speed (transient energy → spatial burst)
- Gate from NDLR or BeatBuddy → particle launch event (each gate = new ball launched)

**Outbound cljseq → CV (for completeness):**
The ES-3 outputs can also carry CV generated by cljseq (via DC-coupled audio samples sent over ADAT). Spatial Field particle position as CV → ES-3 → modular VCAs → direct hardware quad steering. This bypasses the OSC mixer entirely, routing audio through the modular's own VCA matrix rather than a software mixer — potentially lower latency and directly controllable by the modular's signal processing.

### The Feedback Loop

The complete loop unifies the room simulation, the speaker array, and the modular's CV environment:

```
┌─────────────────────────────────────────────────────────┐
│                    cljseq                               │
│  Spatial Field (particle physics + position)            │
│       ↓ VBAP gains                                      │
│  OSC → quad mixer → 4 speakers                          │
└─────────────────────────┬───────────────────────────────┘
                          │
          acoustic energy in the room
                          │
          modular signal path picks up:
          - envelope follower (amplitude → CV)
          - gate detector (transient → gate)
          - joystick (performer intent → CV)
                          │
                    ES-6 CV input → ADAT
                          │
┌─────────────────────────▼───────────────────────────────┐
│              cljseq sidecar bridge                       │
│  DC-coupled audio → normalized value → OSC              │
│       ↑                                                  │
│  Spatial Field parameter update                         │
└─────────────────────────────────────────────────────────┘
```

**What this enables as a performance system:**

The Spatial Field positions audio events in the room. The acoustic energy those events produce feeds back via the modular's envelope followers as CV, which returns to cljseq and adjusts the spatial model. If a dense passage creates a lot of acoustic energy, the envelope follower CV rises, which (by mapping) increases ball speed or launches more particles — the density becomes self-reinforcing. Or mapped inversely: high acoustic energy → increased damping → the system self-limits. The choice of CV→parameter mapping determines whether the system is **generative** (positive feedback) or **self-regulating** (negative feedback).

This is the note-event and spatial equivalent of an analog feedback patch — the room is the feedback path.

### Binaural Mode

Binaural encoding virtualizes the quad (or 3D) space over headphones using Head-Related Transfer Functions (HRTFs). The same Spatial Field position data that drives the quad gains can alternatively be sent to a binaural encoder.

**Output path:**

```
Spatial Field position (x, y, z)
   → azimuth = atan2(x, y) × (180/π)       ; degrees
   → elevation = asin(z) × (180/π)
   → OSC /binaural/azimuth   azimuth-value
      OSC /binaural/elevation elevation-value
   → SuperCollider binaural panner (Ambisonics B-format → binaural decode)
     or Bitwig binaural plugin (OSC-automated)
   → stereo headphone output
```

SuperCollider's Ambisonics toolkit (ATK or IEM plugins) supports binaural decoding from first-order or higher-order Ambisonics. The Spatial Field position encodes directly as an Ambisonics source direction — no per-speaker gain computation needed, the HRTF convolution handles it.

**Binaural vs. Quad as session modes:**

```clojure
(defspatial-field ambient-diffusion
  :room    {:type :pentagon :sides 5.3}
  :output  :quad            ; or :binaural, :ambisonics-foa
  :quad    {:fl :ch1 :fr :ch2 :rl :ch3 :rr :ch4
            :osc-target "osc://localhost:8000"}
  :binaural {:osc-target "osc://localhost:57120"   ; SuperCollider
             :hrtf       :kemar})                  ; or :listen :sadie
```

### Quad as the Natural Format for this Context

Binaural and quadraphonic both predate Atmos by decades and have a strong lineage in electronic/experimental music. For a live modular performance context:
- **Quad** is the right choice: physical speakers create real acoustic events in the room; the performer is inside the sound field; the modular's envelope followers respond to real acoustic energy
- **Binaural** is the right choice for recorded/headphone distribution of the same session

The architecture supports both from the same Spatial Field position data — output mode is a session configuration, not an architectural difference. A recording could capture the quad output for acoustic documentation and simultaneously encode binaural for headphone release.
